Phase 1

Generating User Stories

The Story Agent converts product briefs and discovery notes into structured user stories with acceptance criteria.

AS A → New User

I want to sign up with my email and a password

SO THAT → I can access my personalised dashboard without friction.
AS A → Returning Customer

I want to view my order history filtered by date

SO THAT → I can quickly find past purchases and track refunds.
AS A → Admin

I want to export user data as a CSV

SO THAT → I can run offline analyses without engineering support.

🤖 Story Agent Inputs

  • ▸ Product brief / PRD
  • ▸ Persona definitions
  • ▸ Business goals
  • ▸ Existing Jira epics

✅ Story Agent Outputs

  • ▸ Structured user stories
  • ▸ Story point estimates
  • ▸ Priority ranking
  • ▸ Draft acceptance criteria

Phase 4

Technical Spec Generation

The Spec Agent writes detailed technical design documents from stories and Figma assets.

# Spec: POST /auth/register # Generated: 2026-03-02 | Jira: PROJ-43 endpoint: method: POST path: /api/v1/auth/register auth: none request: body: email: string # RFC-5322, unique password: string # min 8 chars, 1 upper, 1 digit name: string # 2–80 chars responses: 201: body: { userId, email, createdAt } 400: body: { errors: FieldError[] } 409: body: { error: "EMAIL_TAKEN" } side_effects: - send welcome email (async) - emit user.registered event security: - rate_limit: 5 req/min per IP - hash: bcrypt(12)

📄 Spec Sections Generated

  • ▸ Overview & context
  • ▸ API contract (OpenAPI)
  • ▸ Data models & schemas
  • ▸ Component interface (TS types)
  • ▸ Error catalogue
  • ▸ Security considerations
  • ▸ Performance targets

🔗 Living Document

Spec is auto-updated when Jira tickets are refined or designs change. Diffs are posted as PR comments.

📐 Output Formats

Markdown, OpenAPI 3.1, JSON Schema — consumed directly by downstream agents and engineers.

Phase 5

Scenarios & Acceptance Tests

The QA Agent generates BDD scenarios from stories and specs — ready for Cypress, Playwright, or Jest.

Feature: User Registration · Story: PROJ-42
Scenario: Successful sign-up with valid credentials
Given

the user is on the /register page

When

they enter a valid email and strong password

And

they submit the form

Then

they are redirected to /dashboard

And

a welcome email is sent

Scenario: Duplicate email error
Given

the email already exists in the system

When

they submit the form with that email

Then

an inline error: "This email is already registered" appears

And

no account is created

🧪 Scenario Types Generated

  • ▸ Happy path scenarios
  • ▸ Edge cases & boundary values
  • ▸ Error & failure states
  • ▸ Permission / role scenarios
  • ▸ Accessibility checks
  • ▸ Performance thresholds

🤖 Auto-Test Scaffolding

Scenarios are transformed into runnable Playwright or Cypress test files. Engineers just fill in selectors.

🔗 Jira AC Mapping

Each scenario maps to a Jira acceptance criterion. Test pass/fail status synced back to the ticket.

Worked Example

Feature: "Forgot Password" Flow

See how the full pipeline runs for a single feature request in under 15 minutes.

PM drops brief in Slack → Orchestrator triggered

"Users can't reset their password. Add forgot-password flow."

Story Agent → 3 user stories created

Forgot password, reset link, password change — with full AC.

Design Agent → Figma frames linked

Tokens extracted: modal, input states, CTA button variants.

Jira Agent → Epic PROJ-55 + 6 tickets

Frontend, backend, email template tasks auto-created & estimated.

Spec Agent → Technical design doc

POST /auth/forgot-password, JWT reset token, email TTL 1h.

QA Agent → 11 BDD scenarios + test files

Happy path, expired token, invalid email, rate-limit scenarios.

✅ Human review checkpoint → Approved

PM, designer, and lead engineer sign off in Slack thread.

Code Agent → Implementation PR opened

All tests green, Jira tickets moved to In Review automatically.
